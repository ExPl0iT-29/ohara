## Context

Library is single-user, on-device, realistically hundreds not millions of rows. No search or freeform-tag concept exists yet. `Content.topics` (AI-generated) already exists and is stored but never surfaced in the UI. The `feed-organization` change established the pattern for additive nullable/JSON columns (`archivedAt`) and query-layer filtering (`listContentRows` WHERE clauses) — this change follows the same shape for `tags`.

## Goals / Non-Goals

**Goals:**
- Find a saved item by typing a few words, matched against title/summary/extractedText.
- Let the user tag items freely, independent of `contentType`.
- Surface AI-generated `topics` as tappable filter chips alongside user tags — zero new AI calls, just UI on existing data.

**Non-Goals:**
- No ranked/fuzzy search, no stemming — substring match is enough at this library size.
- No tag autocomplete/management UI (rename/merge tags) — add/remove per item only.
- No AI-driven tag suggestion beyond what `enrichContent` already produces as `topics`.

## Decisions

- **Search: `LIKE '%query%'` across `title`, `summary`, `extractedText`, not FTS5.** FTS5 availability in `expo-sqlite`'s bundled SQLite build is not guaranteed across Expo SDK versions/platforms, and setting it up requires a virtual table + insert/update triggers to stay in sync — real complexity for a library this size. Ladder step 1 (does this need to exist) says no: a `LIKE` scan over a few hundred rows is instant on-device. Revisit only if the library grows to a size where this measurably lags (ladder step 4: upgrade when the simple thing actually falls short).
- **`tags: string[]`, JSON-serialized column, same shape as the existing `topics` column.** Reuses the exact pattern `contentRepository.ts` already has for `topics` (`JSON.parse`/`JSON.stringify` in `rowToItem`/`updateContentRow`) — no new serialization concept.
- **Tag filter matches against both `tags` and `topics`.** From the user's perspective these are both "labels on this item" — AI-generated ones just start pre-filled. The filter chip row is generated by unioning distinct values from both columns across the current (non-archived, per existing default) result set; clicking a chip does a `LIKE '%"<value>"%'` match against the serialized JSON column (simple, no new join table, consistent with how `contentType`/`status` filters already work as single WHERE clauses).
- **Search and tag filter compose with the existing Active/Archived toggle**, not replace it — same query-parameter-driven approach as `listContentRows`.

## Risks / Trade-offs

- [`LIKE` search doesn't rank relevance or handle typos] → acceptable at personal-library scale; revisit if it becomes annoying in practice.
- [Tag/topic filter via JSON-column `LIKE` match is a substring match on serialized JSON, not a real containment check] → values are short plain words/phrases from user input or AI output, collision risk (e.g. tag "AI" matching inside topic "AIrflow") is low; mitigate by matching `"<value>"` with quotes included (exact JSON-string-token match) rather than a bare substring.
